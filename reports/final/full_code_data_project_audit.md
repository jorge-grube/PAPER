## E. Code integrity status

**Every active script computes the correct quantities:**

- Turnover formula: `(1/2) Σ |Δw_i|` half-absolute-sum convention. ✓
- TC charge: `gross − rate × TO.shift(1)` (lagged turnover matches 1-week implementation lag). ✓
- Annualised TO = average weekly TO × 52, which equals per-rebalance TO × 13 for the 4-weekly schedule. ✓
- Sharpe = excess return / excess std × √52. ✓
- HMM walk-forward refits every 4 weeks from `MIN_TRAIN_OBS=156`; canonical permutation by VIX z-score. ✓
- HICP-lag6 robustness correctly shifts the raw HICP gap by 6 weeks and recomputes the z-score. ✓
- Statistical tests use HAC/Newey-West (lag=13) and circular block bootstrap (block=13, n=5,000, seed=42). ✓

**Code-hygiene issues:**

- `_port`, `_weekly_to`, `compute_metrics`, etc. helpers are duplicated 7-9× across scripts (P3-06).
- `src/optimization/cvar.py:CVaRConfig.max_weight = 0.35` default vs documented 0.25 (P1-05).
- Dead code: `run_cvar_backtest`, `weighted_cvar_weights`, `RegimeConstraints` in cvar.py (P1-10, P2-13).
- Magic constants (ALPHA, MAX_WEIGHT, REBALANCE, MIN_HISTORY, SCENARIO_CAP, CASH_COL...) duplicated across scripts (P3-05).

## Final note

**The empirical conclusions of the paper are correct and supported by the data.** Specifically:

- Static CVaR remains the most robust benchmark (0.530 Sharpe, 21.4% turnover).
- Naive regime filtering (Regime CVaR-A) fails primarily through turnover (225.8% annually, dropping net Sharpe to 0.346).
- Implementation-aware variants (TC-aware τ=0.10, RC-CVaR, ZEW+λ) improve net Sharpe to ~0.49-0.57 but do not surpass Static CVaR.
- Sovereign FI-expansion improves drawdown (−25.3% → −14.6% in Panel B) but introduces 2022 rate-shock sensitivity (Static CVaR Sharpe −0.026 in Panel B).

These match the paper's stated conclusions verbatim. **The paper is honest, well-documented, and ready for feedback.** The audit findings are reproducibility and clarity issues, not methodological errors.
